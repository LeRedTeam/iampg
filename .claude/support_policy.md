# Support Policy

## Support Philosophy

Support is a cost center. Minimize it through good design, not staffing.

---

## Support Channels

### Free Tier
- GitHub Issues (bugs only)
- GitHub Discussions (community Q&A)
- No guaranteed response time

### Paid Tier
- Email support
- 48-hour response time (business days)
- GitHub Issues with priority label

---

## Support Scope

### In Scope
- Bug reports with reproduction steps
- Documentation clarifications
- Feature requests (no commitment to implement)
- License key issues

### Out of Scope
- AWS IAM consulting
- Policy design advice
- Debugging user's application code
- Integration help beyond documentation
- On-call support

---

## Support Minimization Rules

Every feature must reduce support, not increase it.

### Design principles:
- Clear error messages with fix suggestions
- Deterministic, reproducible behavior
- Explicit failure modes
- Minimal configuration
- Self-documenting CLI help

### Before shipping:
- [ ] Can a user self-serve with docs?
- [ ] Are error messages actionable?
- [ ] Is behavior deterministic?
- [ ] Are edge cases documented?

---

## Escalation Policy

No escalation beyond email support.
No phone support.
No video calls.
No screen sharing.

If an issue can't be resolved via async text:
1. It's probably out of scope
2. Or it's a bug that needs fixing

---

## SLA

No SLA. Response time is a goal, not a guarantee.

Paid users get priority, not guarantees.

---

## Support Metrics to Track

- Tickets per 100 users (lower is better)
- Resolution time (lower is better)
- Repeat questions (signals documentation gaps)
- Feature requests (signals roadmap input)
